# 통계·대시보드 모바일 UI 개선 계획

작성일: 2026-05-07
대상: `apps/stats/templates/stats/index.html`, `apps/stats/templates/stats/life_feedback.html`, `apps/dashboard/templates/dashboard/index.html`

## 배경

모바일 화면에서 다음 UX 마찰이 보고됨:

- 통계 페이지의 **일간/주간/월간/태그분석 탭**이 한 번에 하나만 보여 좁은 화면에서 잦은 탭 전환이 필요
- **목표 진행 카드** 3개(일/주/월)가 항상 펼쳐져 있어 모바일에선 세로 공간을 크게 차지
- **라이프 피드백 토스트**가 wrap으로 도배되어 화면을 덮음 (피드백 5+ 시 심각)
- 대시보드의 **그리드 → 사이드바 태그 선택** 동선이 모바일에서 스크롤 다운 필요

## 작업 항목

### #1 통계 4개 섹션 모바일 단일 컬럼 표시
- **현재**: Bootstrap nav-tabs (data-bs-toggle), 한 번에 1개 탭만 표시
- **목표**: 데스크톱(≥768px)은 그대로 유지. 모바일(<768px)에선 탭 바 숨김 + 4개 `.tab-pane`을 모두 펼쳐서 위→아래 일렬 표시
- **방식**: CSS-only (`@media (max-width: 767.98px)` + `.tab-pane { display: block !important; opacity: 1; }`)
- **부수**: 각 섹션 사이 구분 헤더(daily/weekly/monthly/tag analysis)를 모바일에서 인라인으로 표시 (탭 라벨 재활용)
- **테스트**: 렌더 검증 (모든 섹션이 모바일 viewport에서 모두 보이는지 — 통합 테스트 어려우므로 클래스 존재 여부만)

### #2 목표 진행 일/주/월 accordion + 빈 상태 간략화
- **현재**: 3개 카드 항상 펼침, 등록된 목표 없으면 "등록된 목표가 없습니다" 한 줄
- **목표**:
  - Bootstrap accordion(`.accordion`)으로 묶고 **주간만 기본 펼침**
  - 헤더에 카운트 뱃지: `주간 (3)` / `일간 (0)`
  - 카운트 0이면 펼쳐도 "등록된 목표 없음 → [추가하기]" 한 줄(기존 mypage 링크)
- **테스트**: 사용자 목표가 있을 때 / 없을 때 두 케이스 렌더 검증

### #3 라이프 피드백 다수 시 첫 N개 + 더보기
- **현재**: 모든 피드백 토스트를 `flex-wrap`으로 가로 나열
- **목표**:
  - severity 우선순위(`warning > positive > info`)로 정렬 후 **첫 3개만 표시**
  - 4개 이상이면 하단에 `[+N개 더보기]` 버튼 → 클릭 시 나머지 노출
  - 정렬은 view 단에서 처리 (템플릿 렌더 비용 줄이고 안정적)
- **테스트**: 0/1/3/5개 케이스 — view에서 정렬·슬라이스 정확도

### #4 대시보드 모바일 — 드래그 후 태그 bottom-sheet
- **현재**: 그리드 옆 `#quickInputSidebar`에서 태그 클릭. 모바일에선 `col-lg-4`가 그리드 아래로 떨어져 스크롤 필요
- **목표**:
  - 모바일(<768px)에서 슬롯 선택(touchend / mouseup) 완료 시 화면 하단에 bottom-sheet 슬라이드업으로 **태그 팔레트만** 노출
  - 태그 클릭 → 즉시 저장 또는 확인. 닫기 버튼/배경 탭으로 dismiss
  - 데스크톱은 기존 사이드바 그대로 유지
- **방식**: 기존 `#quickInputSidebar`의 태그 컨테이너 부분을 모바일에선 `position: fixed; bottom: 0` 으로 이동 + `transform: translateY(100%)` ↔ `0` 토글
- **테스트**: 모바일 viewport에서 슬롯 선택 시 sheet 표시 클래스 토글 검증 (JS 단위 테스트는 어려우므로 DOM 구조만 회귀)

## 우선순위 & 일정

| 순서 | 항목 | 작업량 | 종속성 |
|---|---|---|---|
| 1 | 통계 4섹션 모바일 단일 컬럼 | S · 1시간 | 없음 |
| 2 | 목표 진행 accordion + 빈 상태 | M · 2-3시간 | 없음 |

| 4 | 대시보드 모바일 태그 bottom-sheet | L · 4-5시간 | 기존 사이드바 JS 분석 필요 |

각 항목 완료마다:
1. 회귀 테스트 추가
2. 사용자에게 검토 후 커밋 명령 제시
3. 다음 항목 착수 전 검토 단계 거침

## 위험 요소

- **#1**: `display: block !important` 로 강제 노출 시 `.fade` 클래스 transition 충돌 가능. `.tab-pane`의 `opacity: 0` → `1` 명시 필요
- **#2**: accordion 변경으로 기존 `user_goals_*` 컨텍스트 변수 그대로 사용 — 백엔드 영향 없음 예상
- **#3**: severity 정렬 시 view에서 sorted() 사용 → repository/use_case 추가 변경 가능성 있음 (기존 구조 확인 후 진행)
- **#4**: 가장 복잡. 기존 `dashboard.js`의 슬롯 선택/저장 로직 분석 후 진행. 데스크톱 동작 회귀 주의
